## 1. 阶段目标（Phase Goals）

### Phase 0：基线冻结与范围确认（0.5 天）
- 冻结当前线上可运行版本（打 tag）
- 确认“先迁移再切流”的策略，不做一次性大爆炸重写
- 确认需求口径：
  1) 手机尾号查询
  2) 押金退还状态（同尾号多单仅收一次押金）
  3) 月/日数据导出
  4) 错票备注（打印跳号）
  5) 打印时间精确到秒

### Phase 1：单仓改造与工程底座（1~2 天）
- 建立 monorepo（`apps/backend` + `apps/frontend`）
- 统一根命令：`dev/build/test/lint`
- 修复路径、环境变量、部署脚本（PM2/CI）
- 去除前端硬编码 `127.0.0.1` 地址

### Phase 2：后端渐进重构（Express -> NestJS）（3~5 天）
- 建立 NestJS 基础工程与模块边界
- 按模块迁移接口（保留 `/admin/*` 兼容层）
- 安全增强：鉴权、参数校验、密码哈希、CORS 白名单、统一异常处理

### Phase 3：前端重构（CRA/MobX -> Vite/React/TS/Tailwind/Sass）（3~5 天）
- 搭建 Vite + React + TS
- 统一 API 层、路由与状态组织
- 构建基础设计 token 与组件规范（AntD v5 + Tailwind + Sass Modules）

### Phase 4：5 个业务需求落地（2~4 天）
- 双端联动完成 5 个需求与回归
- 保证口径一致（列表、打印、导出）

### Phase 5：验收与切流（1~2 天）
- 功能验收、安全验收、回滚演练
- 分批切流，观测稳定后下线旧实现

## 4. 风险清单（Risk Register）

1) **迁移期接口兼容风险（高）**
- 风险：前端调用与 Nest 新接口不一致
- 缓解：保留 `/admin/*` 兼容层 + 契约测试

2) **打印链路回归风险（高）**
- 风险：打印模板/驱动在新前端下异常
- 缓解：保留旧打印流程开关，先灰度

3) **数据口径不一致风险（中）**
- 风险：列表、导出、打印统计口径不同
- 缓解：统一后端聚合口径，前端只展示

4) **押金规则误判风险（中）**
- 风险：同尾号跨天/重号场景误关联
- 缓解：引入 `depositKey`，规则可配置，补人工修正入口

5) **工期膨胀风险（中）**
- 风险：一次性重写范围过大
- 缓解：严格分阶段，先兼容、后替换

## 6. 回滚策略（Rollback）

### 代码回滚
- 保留旧 Express 与旧前端稳定分支（`legacy/*`）
- 新旧实现双轨运行，按路由/域名切流

### 数据回滚
- 新字段仅追加（不破坏旧字段）
- 上线前做全量备份；迁移脚本支持幂等与反向脚本

### 发布回滚
- PM2 保留上一个稳定进程配置
- 出现 P0 故障可在 5~10 分钟内回切旧服务
